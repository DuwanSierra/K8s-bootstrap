# Paso 40: Install Secrets

## Objetivo
Crear el secret `cni-results-git-credentials` en el namespace `cni-test` usando Terraform, recibiendo `GIT_USER` y `GIT_TOKEN` desde `00.bootstrap`.

## Archivos involucrados
- `40.install-secrets/main.tf`
- `40.install-secrets/variables.tf`
- `40.install-secrets/outputs.tf`
- `40.install-secrets/Makefile`
- `00.bootstrap/variables.tf`
- `00.bootstrap/main.tf`
- `00.bootstrap/terraform.tfvars`

## 1. Definir credenciales en bootstrap
Agregar en `00.bootstrap/variables.tf`:
- `cni_results_git_user` (sensitive)
- `cni_results_git_token` (sensitive)

Cargar valores en `00.bootstrap/terraform.tfvars`:
- `cni_results_git_user = "<tu_usuario_git>"`
- `cni_results_git_token = "<tu_token_pat>"`

## 2. Crear el módulo 40.install-secrets
En `40.install-secrets/main.tf` se define el recurso:
- `kubernetes_secret_v1.cni_results_git_credentials`

Detalles de implementación:
- `metadata.name = "cni-results-git-credentials"`
- `metadata.namespace = "cni-test"`
- `type = "Opaque"`
- Claves en `data`:
  - `GIT_USER`
  - `GIT_TOKEN`

## 3. Encadenar el módulo desde bootstrap
En `00.bootstrap/main.tf` se agrega:
- `module "install_secrets"`
- `source = "../40.install-secrets"`
- Variables de entrada:
  - `git_user  = var.cni_results_git_user`
  - `git_token = var.cni_results_git_token`

Se mantiene el flujo secuencial con:
- `count = local.argocd_ready ? 1 : 0`
- `depends_on = [module.argocd_apps]`

## 4. Aplicar los cambios
Desde `00.bootstrap`:

```bash
make init
make plan
make apply
```

## 5. Verificar en clúster
Comprobar que el secret existe:

```bash
kubectl -n cni-test get secret cni-results-git-credentials
```

Inspeccionar las claves esperadas:

```bash
kubectl -n cni-test get secret cni-results-git-credentials -o jsonpath='{.data}'
```

## Notas operativas
- Este diseño reemplaza el comando manual:
  `kubectl -n cni-test create secret generic cni-results-git-credentials ...`
  por una definición declarativa en Terraform.
- `sensitive = true` evita mostrar valores en salidas normales de Terraform, pero el estado de Terraform sigue conteniendo el valor del secret.
- Si el namespace `cni-test` no existe al momento del `apply`, Terraform fallará hasta que ese namespace esté creado.

## Comportamiento en destroy
Para evitar que `terraform destroy` intente borrar el secret del paso 40 (y falle cuando el cluster ya no esta disponible), `00.bootstrap/Makefile` ahora ejecuta:

```bash
terraform state rm module.install_secrets || true
```

antes del `terraform destroy -auto-approve`.

Esto hace que el secret quede fuera del estado durante el destroy de bootstrap.
